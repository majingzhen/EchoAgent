# EchoAgent 改进计划

> 定位调整：面向开发者和营销从业者的**开源自托管 AI 用研工具**，不做 SaaS，不做多租户。
> 目标：让任何人都能用自己的 LLM API Key 部署一套完整的 AI 虚拟用研工作台。

---

## 一、现有问题清单与解决方案

### P0 — 影响核心可用性

---

#### 问题 1：数据孤立，画像无记忆

**现状**
每次焦点小组 session 独立存在，同一组画像在不同 session 之间没有任何记忆。同一个虚拟消费者第一次说「我不接受超过 200 元的护肤品」，下次提问时他什么都不记得了。

**影响**
- 无法模拟「消费者认知随传播演进」的过程
- 多轮焦点小组结果不连贯，无法积累洞察
- 品牌调研无法在同一组画像上持续深挖

**解决方案**
在 SQLite 中新增 `persona_memory` 表，记录每个画像在每次 session 中的关键表态：

```sql
CREATE TABLE persona_memory (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    persona_id INTEGER NOT NULL,
    session_id INTEGER NOT NULL,
    session_type TEXT NOT NULL,      -- focus_group / simulation
    memory_type TEXT NOT NULL,       -- opinion / preference / reaction
    content TEXT NOT NULL,           -- 记忆内容（简短摘要）
    topic TEXT,                      -- 话题标签
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

焦点小组提问时，先查询该画像过去的相关记忆，拼入 system_prompt：

```
你是 [姓名]，[基本背景]。
你的历史观点记录：
- [2025-03-01] 关于精华液定价：你表示不接受超过 200 元
- [2025-03-05] 关于小红书内容：你认为成分党博主更可信
请保持观点连贯性作答。
```

**工作量**：中（新增 repository + service 方法，修改 `_build_persona_prompt`）

---

#### 问题 2：传播推演数值不可信

**现状**
模拟引擎用 LLM 生成每轮的互动率、情绪分等数值，这些数字是 LLM 编造的，和真实平台数据没有关联。前端图表展示这些数字会误导用户做出错误决策。

**影响**
- 「互动率 12%」这类输出没有参考价值
- 用户把 AI 的想象当作市场预测
- 损害产品可信度

**解决方案**

分两步：

**Step 1（短期）：在 UI 上明确标注「参考性数据」**
在传播推演结果页面所有数值旁加免责说明，把指标从「绝对数值」改为「相对趋势」展示（用高/中/低代替具体百分比）。

**Step 2（中期）：引入基准校准参数**
在创建模拟时，允许用户输入品类基准数据（如「该品类小红书平均互动率 3-8%」），模拟引擎将基准注入 prompt，生成的数值在合理区间内。

```python
# SimulationConfig 新增字段
benchmark_engagement_rate: float | None = None   # 品类基准互动率
benchmark_sentiment_score: float | None = None   # 品类基准情绪分
```

**工作量**：Step 1 小，Step 2 中

---

#### 问题 3：焦点小组画像各自独立，缺乏动态交互

**现状**
提一个问题，6 个画像各自独立作答，互不影响。真实焦点小组的价值恰恰在于「甲说了 A，乙听完 A 后反驳，丙被说服改变立场」这种动态碰撞。

**影响**
- 缺失焦点小组最核心的洞察来源：观点碰撞和立场演变
- 结果只是 6 份独立问卷，不是真正的小组讨论

**解决方案**
在现有「独立作答」基础上，新增可选的「讨论模式」：

```
Round 1：所有画像独立作答（现有逻辑）
Round 2：AI 模拟主持人提炼分歧点
Round 3：持不同意见的画像互相回应，其他画像表明立场是否改变
Round 4：AI 汇总「共识」「持续分歧」「立场转变者」
```

前端新增 `mode: "independent" | "discussion"` 切换，discussion 模式展示对话流程而非并列卡片。

**工作量**：大（需要重写 `_run_ask` 的核心流程）

### P1 — 影响产品深度

---

#### 问题 4：内容工坊 A/B 测试没有闭环

**现状**
工坊生成 A/B 两版文案，但没有任何机制记录「哪版实际上线了」「上线后效果如何」。A/B 测试功能是孤立的，没有反馈回路。

**影响**
- 系统无法从实际结果学习
- 用户每次都要重新输入背景，无法复用经验

**解决方案**
新增 `content_result` 表记录文案上线结果：

```sql
CREATE TABLE content_result (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    workshop_session_id INTEGER NOT NULL,
    variant TEXT NOT NULL,               -- A/B/C
    platform TEXT,
    actual_engagement_rate REAL,         -- 用户填写的实际互动率
    actual_conversion_rate REAL,         -- 实际转化率（可选）
    notes TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

工坊结果页面新增「记录上线结果」入口，填写后：
1. 在该 session 标注哪版胜出
2. 在画像组页面聚合：「该画像组历史评分 vs 实际结果的偏差」，让用户判断这组画像的预测准确度

**工作量**：中

---

#### 问题 5：市场图谱可视化对用户无实用价值

**现状**
市场图谱生成知识图谱并做可视化，但用户真正需要的是「竞品有哪些短板」「消费者在哪些方面不满意竞品」这类直接结论，而不是一张节点-关系图。

**影响**
- 功能视觉复杂但结论提取困难
- 知识图谱可视化对非技术用户完全不友好

**解决方案**
将市场图谱的输出重心从「图谱」转移到「竞品分析报告」：

新的输出结构：
```
1. 竞品核心痛点 Top 5（带来源引用）
2. 消费者最关注的维度（价格 / 成分 / 效果 / 包装 / 服务）
3. 竞品弱点可被利用的角度（直接对接内容工坊 Brief）
4. 附：原始知识图谱（折叠，给有需要的用户）
```

**工作量**：中（修改 `market_service` 的输出结构，前端重写展示组件）

---

#### 问题 6：工作流编排复杂度大于价值

**现状**
工作流把 7 个模块串联自动执行，但用户实际使用时需要在每步「看结果 → 决定要不要继续 → 调整参数」，全自动流水线剥夺了这个控制权，反而降低了结果质量。

**影响**
- 自动化执行的中间结果质量低（因为没有人工干预优化每步的输入）
- 开发维护成本高（复杂的步骤状态机）
- 用户几乎不会使用全自动模式

**解决方案**
将工作流从「自动执行」改为「引导式流程」：

不自动执行各步骤，而是提供一个**推荐路径面板**：
```
你正在做：精华液小红书投放
推荐流程：
① 画像工厂 → ② 市场图谱 → ③ 内容工坊 → ④ 传播推演
[每步完成后，点击「下一步」跳转，自动携带上一步的关键参数]
```

保留工作流 session 的记录和跳转，去掉自动执行逻辑。

**工作量**：中（重写前端，简化后端）

### P2 — 开源工具专项优化

---

#### 问题 7：部署配置对新用户不友好

**现状**
配置分散在 `config/app.yaml`，没有环境变量示例，docker-compose 缺失，初始化数据库需要手动执行 SQL。

**解决方案**
- 提供 `.env.example` 模板，所有必填配置有注释说明
- 提供 `docker-compose.yml`：一键启动 backend + frontend
- 启动时自动检测并初始化数据库（`alembic` 或 startup 事件里执行 `CREATE IF NOT EXISTS`）
- `README.md` 提供 5 分钟快速启动指引

**工作量**：小-中

---

#### 问题 8：LLM 提供商绑定过深

**现状**
代码中大量假设了 DashScope/Qwen 的 API 格式（兼容 OpenAI），但没有清晰的切换文档，用 OpenAI、Claude、本地 Ollama 时需要用户自行摸索。

**解决方案**
在 `config/app.yaml` 中明确列出各提供商的配置示例（注释形式），`LLMClient` 启动时打印当前使用的模型信息方便用户确认：

```yaml
llm:
  # --- Qwen (DashScope) ---
  api_key: sk-xxx
  base_url: https://dashscope.aliyuncs.com/compatible-mode/v1
  model_name: qwen-plus

  # --- OpenAI ---
  # api_key: sk-xxx
  # base_url: https://api.openai.com/v1
  # model_name: gpt-4o

  # --- Ollama (本地) ---
  # api_key: ollama
  # base_url: http://localhost:11434/v1
  # model_name: qwen2.5:14b
```

**工作量**：小（主要是文档和配置模板）

## 二、功能取舍建议

### 建议移除或降级

| 功能 | 建议 | 理由 |
|------|------|------|
| 工作流全自动执行 | 改为引导式路径（见问题 6） | 自动化反而降低质量 |
| 画像组合推荐 | 移除 | 用户知道自己的目标人群，AI 推荐可信度低 |
| 传播推演对比 | 移除 | 两次模拟的可比性本身就不成立 |
| 知识图谱可视化 | 折叠为附件 | 用报告替代图谱（见问题 5） |
| 多租户 tenant_id | 硬编码固定值，移除相关字段 | 开源单用户工具无需此字段 |

### 建议新增

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 画像记忆 | P0 | 见问题 1 |
| 焦点小组讨论模式 | P0 | 见问题 3 |
| A/B 测试结果回填 | P1 | 见问题 4 |
| 模板库 | P1 | 保存常用画像组、文案结构、Brief 模板，复用经验 |
| 一键部署（Docker） | P1 | 开源工具的基础门槛 |
| 多 LLM 快速切换 | P2 | 开源工具的核心诉求之一 |

## 三、执行优先级

```
Sprint 1（核心可用性）
├── 画像记忆持久化
├── 焦点小组讨论模式
└── 传播推演数值免责标注

Sprint 2（产品深度）
├── 市场图谱输出重构（图谱→报告）
├── A/B 测试结果回填
└── 工作流改为引导式路径

Sprint 3（开源就绪）
├── Docker 部署方案
├── .env.example + 配置注释
├── 模板库（画像 / Brief / 场景）
└── README 快速启动文档
```

## 四、不变的核心

以下是产品的核心价值，不做大改：

- **焦点小组**：AI 虚拟消费者对话，这是最大差异化点
- **内容工坊**：多 Agent 流水线生产文案，结构合理
- **舆情哨兵**：危机评估+方案生成，有真实刚需
- **策略参谋**：多模型辩论结构，值得继续打磨

> 围绕这四个核心做深，比继续扩展新模块更有价值。
